# 一、项目背景

随着 NAS 的普及，越来越多用户将图片、影视、小说等资源存储在个人 NAS 中。

但随着资源数量不断增长，用户开始面临新的问题：

* 收藏越来越多
* 消费越来越少
* 找不到想看的内容
* 已有资源被遗忘
* 重复下载和收藏
* 打开 NAS 后不知道看什么

当前主流媒体系统（Jellyfin、Emby、Plex、Calibre 等）主要解决：

* 文件管理
* 元数据管理
* 内容播放

而 My Media Hub 希望解决：

**内容发现（Discovery）问题。**

---

# 二、项目愿景

让 NAS 从：

```text
存储中心
```

升级为：

```text
内容发现中心
```

帮助用户持续从已有资源中发现值得消费的内容。

---

# 三、产品定位

My Media Hub 是一个面向 NAS 的个人媒体发现中心。

它的核心目标不是管理资源，而是帮助用户发现资源。

---

## 产品定位

My Media Hub 是：

* Discovery Engine（发现引擎）
* Recommendation Engine（推荐引擎）
* Personal Media Hub（个人媒体中心）

---

## 产品边界

My Media Hub 负责：

* 媒体索引
* 元数据管理
* 用户行为采集
* 内容发现
* 内容推荐

My Media Hub 不负责：

* 媒体播放
* 媒体下载
* 文件同步
* 文件存储
* 文件管理
* 媒体转码

系统永远聚焦：

```text
Discovery（发现）
```

而不是：

```text
Management（管理）
```

# 五、用户模型

## V1

采用单用户模型。

系统默认存在一个 Profile：

```text
Default Profile
```

所有行为数据均归属于该 Profile。

包括：

* 收藏
* 评分
* 浏览记录
* 搜索记录
* 推荐记录

---

## 不纳入当前范围

暂不支持：

* 用户注册
* 用户登录
* 权限系统
* 多用户隔离

---

## 后续规划

未来支持：

* 多 Profile
* 家庭成员 Profile
* 独立兴趣模型

但不属于当前版本目标。

---

# 六、核心价值

帮助用户：

* 发现被遗忘资源
* 发现高价值资源
* 发现与兴趣相关资源
* 提升资源利用率
* 减少重复收藏
* 提高内容消费率

---

# 七、产品原则

## Discovery First

发现优先。

首页即发现流。

---

## Recommendation Second

推荐其次。

推荐系统是产品核心。

---

## Search Third

搜索第三。

搜索用于辅助发现。

不是产品核心竞争力。

---

## Management Last

管理最后。

标签、分类、元数据等能力均服务于推荐系统。

---

## Behavior Driven

行为驱动。

用户行为是兴趣模型的核心来源。

包括：

* 收藏
* 评分
* 已看
* 搜索
* 点击
* 不感兴趣

系统优先根据行为构建用户兴趣画像。

---

# 八、内容发现闭环

系统核心数据流：

```text
文件扫描
↓
媒体索引
↓
用户浏览
↓
用户行为
↓
兴趣画像
↓
推荐引擎
↓
发现流
↓
用户继续消费
```

形成持续优化的发现闭环。

# 十、功能范围

## 10.1 媒体库

功能：

* 资源扫描
* 元数据索引
* 标签管理
* 分类管理
* 媒体统计
* 媒体详情

---

## 10.2 搜索

功能：

* 关键词搜索
* 标签搜索
* 分类搜索
* 条件筛选
* 排序
* 搜索建议

---

### 搜索架构原则

搜索能力通过 Search Strategy 抽象。

当前实现：

* LIKE
* FTS

未来可扩展：

* Bleve
* Elasticsearch
* Semantic Search

业务层不感知具体实现。

---

## 10.3 用户行为

功能：

* 搜索历史
* 浏览记录
* 收藏
* 评分
* 已看
* 不感兴趣

用户行为是推荐系统最重要的数据来源。

---

### V1

* 随机推荐
* 最近新增

* 标签推荐
* 分类推荐

---

### V2

* 相似推荐

* Interest Profile

---

### V3

* AI Discovery
* 语义推荐

---

## 10.5 发现流

支持：

* 今日推荐
* 猜你喜欢
* 最近新增
* 继续浏览
* 重新发现
* 随机探索

# 十一、不做什么

## 不做播放器

实际播放交由第三方应用。

例如：

* Jellyfin
* Emby
* VLC
* 阅读器

---

## 不做下载器

不负责资源下载。

---

## 不做网盘

不负责文件存储。

---

## 不做媒体转码

不负责媒体处理。

---

## 不做复杂权限系统

当前仅支持单用户模型。

---

# 十二、非功能需求

## 可维护性

系统必须遵循：

```text
API
↓
Service
↓
Repository
↓
Database
```

禁止：

```text
API → SQL
API → Database
Service → SQL
```

---

## 可扩展性

新增媒体类型时：

不得修改核心业务流程。

仅允许：

* 新增 MediaType
* 新增 Metadata Extractor
* 新增 Parser

---

## 可替换性

以下模块必须可替换：

* Search Engine
* Recommendation Strategy
* Metadata Extractor

---

## 稳定性

系统应长期运行于 NAS 环境。

不依赖：

* Docker
* Redis
* Kafka
* RabbitMQ

# 十四、成功标准

用户打开首页后：

无需主动搜索。

即可发现值得消费的内容。

---

衡量指标：

* 5 秒内看到推荐内容
* 30 秒内找到感兴趣内容
* 推荐点击率持续提升
* 用户主要通过推荐而非搜索发现内容
* 已有资源持续被重新消费

# 十六、最终目标

用户打开 My Media Hub 后：

不是去搜索内容。

